# Story 8.1: FileTracker 文件结构设计

Status: done

## Story

As a 系统架构师,
I want 设计 FileTracker 的文件存储结构,
so that 可以用文件系统管理任务状态和详情.

## Acceptance Criteria

1. **AC1: 状态索引文件结构**
   - Given 创建新需求任务
   - When 初始化文件结构
   - Then 生成 `.sym/{task-id}/task.md` 状态索引文件
   - And 使用 YAML frontmatter 存储元数据
   - And 使用 markdown 列表存储子任务状态

2. **AC2: 子任务目录结构**
   - Given 状态索引文件已创建
   - When 创建子任务目录
   - Then 生成 `Planner/` 目录存放 P 类子任务
   - And 生成 `Generator/` 目录存放 G 类子任务
   - And 生成 `Evaluator/` 目录存放 E 类子任务
   - And 每个子任务一个独立详情文件

3. **AC3: 迭代版本管理**
   - Given 迭代发生
   - When 创建新版本子任务
   - Then 文件名包含版本号 (如 G4-代码实现-v2.md)
   - And 迭代历史按时间顺序排列
   - And 状态索引文件记录所有版本

**FRs covered:** NFR4, FR60

## Tasks / Subtasks

- [ ] Task 1: 设计文件结构规范 (AC: 1, 2, 3)
  - [ ] 1.1 定义状态索引文件格式
  - [ ] 1.2 定义子任务详情文件格式
  - [ ] 1.3 定义目录结构规范

## Dev Notes

### 文件结构示例

```
.sym/
└── SYM-001/
    ├── task.md                 # 状态索引
    ├── Planner/
    │   ├── P1-需求澄清.md
    │   ├── P2-BDD规则.md
    │   ├── P3-领域建模.md
    │   ├── P4-架构设计.md
    │   └── P5-接口设计.md
    ├── Generator/
    │   ├── G1-BDD测试脚本-v1.md
    │   ├── G2-集成测试-v1.md
    │   ├── G3-单元测试-v1.md
    │   ├── G4-代码实现-v1.md
    │   ├── G4-代码实现-v2.md  # 迭代版本
    │   └── ...
    └── Evaluator/
        ├── E1-评估验收-v1.md
        ├── E1-评估验收-v2.md  # 迭代版本
        └── ...
```

### 状态索引文件格式 (task.md)

```yaml
---
id: SYM-001
title: 用户认证功能
status: in-progress
phase: generator
iteration: 2
created: 2026-04-07
updated: 2026-04-07T14:30:00
---

# Planner
- P1: 需求澄清 ✅
- P2: BDD规则 ✅
- P3: 领域建模 ✅
- P4: 架构设计 ✅
- P5: 接口设计 ✅

# Generator
- G1: BDD测试脚本-v1 ✅
- G2: 集成测试-v1 ✅
- G3: 单元测试-v1 ✅
- G4: 代码实现-v1 ❌ (失败)

- E1: 评估验收-v1 ❌ (触发迭代)

```
