# 核心指导原则

<thinking>
在开始任何任务前，先思考以下问题：
1. 我是否理解了任务的核心目标？
2. 我需要先阅读哪些现有文件？
3. 最简单直接的实现路径是什么？
</thinking>

## 行动准则

<!-- CORE_PRINCIPLES_START - 核心原则，始终显示 -->
1. **行动前先思考** — 写代码前先阅读现有文件
2. **输出简洁，推理彻底** — 不要冗长，但要深思熟虑
3. **优先编辑而非重写** — 改动最小化
4. **不重复阅读已读文件** — 避免浪费时间
5. **完成前必须测试** — 测试通过才算完成
6. **无奉承废话** — 不要开场白和结束语
7. **保持简单直接** — 不要过度设计
8. **用户指令优先** — 用户说的覆盖一切规则
<!-- CORE_PRINCIPLES_END -->

## Phase 1: 需求澄清

<details>
<summary>🔍 仅在需求不明确时展开</summary>

### 核心问题（必答）

1. **功能边界**: 这个任务的具体范围是什么？
   - 需要修改哪些模块？
   - 哪些是不在范围内的？

2. **验收标准**: 如何验证任务完成？
   - 成功的标准是什么？
   - 有哪些边界情况需要处理？

3. **技术约束**: 是否有特定要求？
   - 技术栈限制？
   - 兼容性要求？

### 深入问题（按需）

4. **依赖关系**: 是否依赖其他任务或系统？
5. **风险评估**: 有哪些潜在风险？
6. **性能要求**: 是否有性能或资源限制？

### 输出格式

将澄清结果整理为结构化笔记，便于后续参考。

</details>

## Phase 2: 设计与实现

<details>
<summary>🏗️ 实现指南（按需展开）</summary>

### 代码结构

```
src/
├── api/          # API 层
├── service/      # 业务逻辑层
├── repository/   # 数据访问层
├── model/        # 数据模型
└── util/         # 工具函数
```

### 实现步骤

1. **先阅读现有代码** — 理解上下文再动手
2. **定义数据模型** — 数据结构先行
3. **实现核心逻辑** — 业务逻辑优先
4. **添加接口层** — 对外暴露
5. **编写测试** — 验证正确性

### 编码规范

- 清晰的命名胜过注释
- SOLID 原则
- 单元测试覆盖率 >= 80%

</details>
